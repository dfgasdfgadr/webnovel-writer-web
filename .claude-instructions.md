# Claude Code 全局指令（NovelCraft）

## 强制规则：单元测试

**每个功能必须有单元测试。** 详见 [`docs/TESTING.md`](docs/TESTING.md)。

- 新功能与 bug 修复：**测试与实现同一 commit**（或测试先行）
- Phase 完成前：运行 `pnpm test` 全绿 + 覆盖率达标
- 交接文档须含「测试验收表」
- Bug 修复必须附带回归测试

---

## 强制规则：阶段交接文档

**每个 Phase 完成后，必须生成交接文档，否则视为阶段未完成。**

### 交接文档要求

1. **文件路径**：`docs/handoffs/PHASE{N}_HANDOFF.md`（N = 0, 1, 2, 3, 4）
2. **模板**：复制 `docs/handoffs/HANDOFF_TEMPLATE.md` 填写，不可省略任何章节
3. **生成时机**：Phase 全部验收项完成后、最终 git commit 之前
4. **Git**：交接文档必须纳入该 Phase 的最后一个 commit

### 交接文档必须包含

- 本阶段目标回顾与交付物清单
- 架构变更摘要（含 API/数据模型变更）
- 验收结果表格（PASS/FAIL/SKIP + 备注）
- 如何运行与手动验证步骤
- 已知问题与技术债（标注优先级与建议处理阶段）
- **下一阶段输入**：首要任务、不要重复做的项、环境注意事项
- 关键文件索引 + 本阶段 Git 提交历史 + Changelog

### Phase 完成检查清单

- [ ] 本 Phase 功能验收全部 PASS 或 SKIP 有说明
- [ ] `EXECUTION_BRIEF.md` 或对应 Phase brief 更新 STATUS=DONE
- [ ] `docs/handoffs/PHASE{N}_HANDOFF.md` 已生成
- [ ] `CLAUDE.md` 中「当前阶段」更新为 Phase {N+1}
- [ ] Git commit 含交接文档

### 开始新 Phase 时

1. **先读** PM 签发的阶段执行简报：`docs/briefs/PHASE{N}_EXECUTION_BRIEF.md`（STATUS 须为 PENDING 或 IN_PROGRESS）
2. **再读**上一阶段交接文档：`docs/handoffs/PHASE{N-1}_HANDOFF.md`
3. **再读**计划文档对应 Phase 章节
4. **不要**重复上一阶段已完成的基建工作

## 强制规则：阶段执行简报（PM → Claude Code 派单）

**每个 Phase 开始前，PM（Cursor）必须先签发执行简报，Claude Code 不得在无简报情况下启动新 Phase。**

### 简报体系

| 文件 | 用途 |
|------|------|
| `docs/briefs/EXECUTION_BRIEF_TEMPLATE.md` | 模板（PM 复制填写） |
| `docs/briefs/PHASE{N}_EXECUTION_BRIEF.md` | 各阶段可执行简报（N=0..4） |
| `EXECUTION_BRIEF.md` | Phase 0 历史简报（已完成，保留归档） |

### PM 工作流（每个 Phase）

1. 读上一阶段 `PHASE{N-1}_HANDOFF.md` 的「下一阶段输入」
2. 读计划文档 §7 Phase {N} + §8 验收标准
3. 撰写/更新 `docs/briefs/PHASE{N}_EXECUTION_BRIEF.md`（STATUS: PENDING）
4. 更新 `docs/CURRENT_TASK.md`、`.claude-run-prompt.txt` 指向该简报
5. 用户确认后，启动 Claude Code 读简报执行
6. Phase 完成后：简报 STATUS → DONE + 生成交接文档

### Claude Code 工作流（每个 Phase）

1. 读 `docs/briefs/PHASE{N}_EXECUTION_BRIEF.md`（第一优先级）
2. 按简报目标逐项实现 + 单测
3. 完成验收自检表
4. 简报 STATUS → DONE
5. 生成 `docs/handoffs/PHASE{N}_HANDOFF.md`
6. 更新 `CLAUDE.md`、`docs/PROGRESS.md`

## 其他规则

- **你是唯一执行者**：bug 修复、功能开发、资料查询、命令执行均由你完成；**PM 只把控进度，不直接改代码**
- PM 发现 bug 时会写入 `docs/CURRENT_TASK.md` 并启动你处理；你负责 fix + 回归测试 + commit
- 当前紧急任务见 `docs/CURRENT_TASK.md`；进度更新到 `docs/PROGRESS.md`
- 前端只用 shadcn/ui + Tailwind，禁止 Ant Design
- 每完成一个大步骤 git commit，message 中文简述 why
- 前端开发遵循 frontend-design 技能 + UI/UX Pro Max + shadcn MCP
- **每个功能必须有单元测试**（见 `docs/TESTING.md`）；`pnpm test` 通过方可 commit Phase 里程碑
